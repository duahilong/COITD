# 跨机器部署教程实机验证记录（测试机）

## 验证时间
- 2026-04-19（Asia/Shanghai）

## 验证目标
- 将测试机上旧部署痕迹清空后，仅按照《跨机器部署教程》重新部署。
- 验证脚本、systemd 定时、Web 页面是否可完整恢复。

## 验证环境
- 服务器：`10.0.0.18`
- 用户：`root`

## 执行过程
1. 清空旧部署
- 停止并禁用旧 timer/service/web service。
- 删除 `/etc/systemd/system/cfst-ddns.*` 与 `/etc/systemd/system/cfst-ddns-web.service`。
- 删除旧运行记录：
  - `/root/coitd/scripts/cfst_ddns/state`
  - `/root/coitd/scripts/cfst_ddns/logs`
  - `/root/coitd/scripts/cfst_ddns_web/ddns_web_server.log`
- 清理后确认：无 timer、无旧 state/log、8091 端口未监听。

2. 重建部署
- 重新安装以下 unit 文件并启用：
  - `cfst-ddns.service`
  - `cfst-ddns.timer`
  - `cfst-ddns-web.service`
- 执行：
  - `systemctl daemon-reload`
  - `systemctl enable --now cfst-ddns.timer`
  - `systemctl enable --now cfst-ddns-web.service`

3. 端到端验证
- 手动触发：`systemctl start cfst-ddns.service`
- 服务执行完成且退出码成功。
- 定时器状态：`active (waiting)`，存在下次触发时间。
- Web 服务状态：`active (running)`。

## 验证结果
1. summary 生成成功
- `state/latest.json` 存在并可读取。
- 最新 `run_id`: `20260419-235221-86ad88`
- `status`: `success`

2. 多 IP 推送能力正常
- `selected_ips` 数量：`2`
- `config.push_ip_count`: `2`
- `line_list`: `telecom,unicom`
- `ddns.operations`：4 条（电信 2 条 + 联通 2 条），全部成功。

3. Web API 正常
- `/healthz` 返回 `up`
- `/api/dashboard` 返回 `ok=true`
- 页面可展示 latest/timer/service 关键状态。

## 发现与修正
- 教程原示例默认写了 `FOLDER=/root/coitd/CloudflareSpeedTest`，但测试机实际 `cfst` 在 `/root/coitd/scripts/cfst_ddns`。
- 已修正文档：明确 `FOLDER` 必须指向“实际包含 cfst 的目录”，并补充重装清理章节。

## 结论
- 部署教程经实机验证可完整落地。
- 在新机器部署时，只需保证：
  1) `FOLDER` 指向真实 `cfst` 目录；
  2) AliDNS 凭据与域名配置正确；
  3) 按教程创建并启用 systemd 单元。
